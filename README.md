## ✅ Assertion Logic

### CLI Assertions

Each assertion in `unified_ksi_validations.json` includes:

- `cli_command`: the AWS CLI used to generate evidence (e.g. `aws ec2 describe-flow-logs`)
- `assertion`: `true` or `false` based on output evaluation
- `assertion_reason`: why the assertion passed or failed
- `validation_timestamp`: UTC ISO-8601 timestamp of validation
- `commit_sha`: Git SHA referencing the evidence commit
- `evidence_path`: relative path to the output file

Assertions are determined by parsing CLI output and matching expected fields or conditions aligned with each KSI’s intent.

> Example: For `KSI-SVC-03`, the AWS CLI command `aws kms list-keys` is expected to return keys that are encrypted and rotated — and the output is checked for compliance.

## 🧭 Evidence Flow

This repository reflects a GitHub-native trust model:

- Evidence is generated on a schedule via GitHub Actions
- Each `.json` output is committed and its Git SHA and UTC timestamp recorded
- `evidence_commit_metadata.json` maps each file to its origin commit
- `unified_ksi_validations.json` includes the evidence path, command, result, and reason

---

## 🛠 Automation and GitHub Actions

Evidence is generated by a single YAML workflow in the private repo. It:

1. Executes all CLI validations across KSI families
2. Writes the raw CLI output to `evidence_v2/KSI-*/raw_output.json`
3. Collects commit metadata
4. Builds `unified_ksi_validations.json`
5. Pushes it here to power the dashboard

This ensures 20x-style continuous assessment and reproducibility.

---


## 🗂 File Structure and Evidence Automation Workflow

### 🔐 Evidence Trace and Git Commit Strategy

All CLI-generated evidence (`evidence_v2/`) is committed via GitHub Actions.

- Each evidence file’s commit SHA and UTC timestamp is extracted using:
  ```bash
  git log -1 --format="%H|%ct" -- "$file"
  ```
- These values are stored in `evidence_commit_metadata.json`, used by the dashboard to verify evidence traceability.
- Because evidence is generated in a single GitHub Action run, it's expected that multiple files share the same SHA and timestamp.

This method ensures auditor-verifiable evidence provenance.

## 📦 Evidence Trust Chain

Every assertion ties back to:

1. A **CLI command**
2. A **raw evidence file**
3. A **commit SHA**
4. A **UTC validation timestamp**

This establishes a fully verifiable chain from command execution ➡️ evidence ➡️ assertion ➡️ dashboard render.

> If any link in the chain is missing or unverifiable, the assertion fails by design.
